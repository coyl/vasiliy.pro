Gestione CORS in Ruby

Uso di Rack::Cors con Sinatra

Quando una pagina web su un'origine (es. https://app.example.com) effettua una richiesta a un'API su un'altra origine (es. https://api.example.com), il browser applica la Same-Origin Policy. Di default le richieste cross-origin vengono bloccate, a meno che il server non le autorizzi esplicitamente tramite CORS (Cross-Origin Resource Sharing).

Il CORS funziona tramite intestazioni HTTP. Il browser può inviare una richiesta di preflight OPTIONS; il server deve rispondere con intestazioni come Access-Control-Allow-Origin, Access-Control-Allow-Methods e Access-Control-Allow-Headers per consentire la richiesta effettiva. In Ruby un modo comune per aggiungere queste intestazioni è la gemma rack-cors: si integra nello stack Rack e inietta per te le intestazioni CORS corrette.

Qui sotto è mostrata una piccola app Sinatra che abilita il CORS per tutte le origini e tutte le route con Rack::Cors. Il middleware viene montato con use Rack::Cors; il blocco allow definisce quali origini sono consentite (origins '*' significa qualsiasi origine), quali risorse (resource '*'), quali intestazioni HTTP (headers :any) e quali metodi (methods :any). Il resto dell'app è una route catch-all che restituisce 200 e un contatore di richieste, utile per verificare che le richieste cross-origin vadano a buon fine.

In produzione evita origins '*' a meno che non ti serva davvero consentire a qualsiasi sito di chiamare la tua API. Preferisci una whitelist: ad es. origins 'https://app.example.com', 'https://admin.example.com'. Puoi anche limitare resource a percorsi specifici e methods ai soli metodi che la tua API usa effettivamente. Le applicazioni Rails possono usare la gemma rack-cors allo stesso modo, oppure configurare il CORS tramite config/initializers/cors.rb quando si usa il supporto CORS integrato. In ogni caso il browser consentirà la richiesta cross-origin solo se le intestazioni CORS del server corrispondono a quanto richiede il client.
